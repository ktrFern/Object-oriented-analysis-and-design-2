# Лабораторная работа №4: Шлюз

## Предметная область
Настольное приложение для террариумистики, где возможно:
- **Управлять данными о животных** - просматривать, добавлять, редактировать и удалять записи о животных в табличном виде;
- **Вести журнал событий** - фиксировать события (кормление, уборка, линька, ветеринар и др.) с привязкой к конкретному животному;
- **Искать записи** - фильтровать животных по имени или виду в реальном времени;
- **Просматривать календарь** - планировщик с отображением событий по дням и месяцам.

## Проблема
Без паттерна приложение работает напрямую с базой данных из класса `App`, встраивая SQL-запросы прямо в класс приложения. Это порождает ряд проблем:

- Один класс `App` одновременно отвечает за построение UI, обработку событий и выполнение SQL-запросов.
- SQL-запросы и строка подключения разбросаны по всему классу. Смена базы данных потребует правки во множестве мест.

## Решение: паттерн Шлюз

Паттерн **Шлюз** инкапсулирует весь SQL-доступ к одной таблице в отдельном классе. Клиентский код работает с объектами предметной области, не зная ничего о структуре базы данных и SQL-синтаксисе.

В проекте реализованы два шлюза: `AnimalGateway` инкапсулирует все операции с таблицей `animals` (добавление, поиск, обновление, удаление, подсчёт), а `EventGateway` - с таблицей `events` (добавление, выборка по дате, по месяцу, по животному, удаление).

Роль **клиента** выполняет класс `GatewayApp`. Он не содержит ни одного SQL-запроса - все обращения к базе проходят через методы шлюзов: `animalGateway.search(val)`, `eventGateway.addEvent(...)`, `animalGateway.deleteAnimal(id)`.

## Обработка связанных таблиц

`AnimalGateway` реализует каскадное удаление вручную. При вызове `deleteAnimal(id)` шлюз удаляет все события животного из таблицы `events`, затем удаляет самого животного из `animals`.

## Два варианта реализации

### С паттерном
`GatewayApp` работает исключительно через объекты `AnimalGateway` и `EventGateway`. Весь SQL сосредоточен в двух классах-шлюзах, каждый из которых отвечает ровно за собственную таблицу. Добавление новой операции с базой данных не затрагивает UI-код - достаточно добавить метод в соответствующий шлюз.

### Без паттерна
`App` содержит приватные методы с SQL прямо внутри класса приложения: `addAnimal()`, `searchAnimals()`, `updateAnimal()`, `deleteAnimal()`, `addEvent()` и т.д. Класс выполняет работу одновременно UI-контроллера и слоя доступа к данным.

## Вывод

Внедрение паттерна Шлюз существенно улучшило архитектуру приложения:

`GatewayApp` занимается исключительно UI, а `AnimalGateway` и `EventGateway` - исключительно работой с базой данных. Каждый класс делает одно дело.

При смене СУБД или схемы таблицы достаточно изменить один шлюз - клиентский код остаётся нетронутым.
